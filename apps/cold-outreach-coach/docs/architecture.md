# Cold Outreach Coach — Architecture

## Overview

Cold Outreach Coach is a **Flutter desktop application** that uses the **AI Market Protocol v2** to discover, purchase, and invoke decentralized capabilities for cold email optimization. The app follows a strict **privacy-first** architecture: email content never leaves the device. Only anonymized structural metrics are sold through the marketplace.

## Five-Phase Marketplace Cycle

The app uses the **AimarketAgent** Dart SDK (`aimarket_agent` package) to implement the AI Market Protocol v2 consumer cycle:

### Phase 1: Discovery

The app sends a natural-language intent to hub.aicom.io. The hub returns a ranked list of capabilities matched to the intent.

**Example discovery requests:**

| Intent | Filters | Expected results |
|--------|---------|-----------------|
| "email deliverability rules for cold outreach Q2 2026" | category=career, budget=5.00 | SPF/DKIM checkers, content heuristics bundles |
| "reply-rate signals for B2B SaaS cold emails" | category=career, limit=10 | Structural benchmark packages |
| "ICP persona templates for fintech" | category=career | Industry-specific template builders |

### Phase 2: Open Channel

A pre-funded payment channel is opened. The user deposits USDT (on Base chain) into a smart contract escrow. This funds multiple invocations without per-call gas costs.

**Channel economics:**
- Typical deposit: $5.00 USDT
- Typical per-call cost: $0.05-$0.50
- Channel expiry: 24 hours (configurable)
- Unused balance: refunded on close

### Phase 3: Invoke

The app sends a capability ID and input payload (structural metrics only — never email content). The hub executes the capability in a TEE (AWS Nitro or Intel TDX) and returns results.

**Privacy boundary:**
```
Allowed input:  {"industry": "saas", "target_role": "VP Engineering",
                 "word_count": 85, "paragraph_count": 4,
                 "has_questions": true, "link_count": 2}
Never sent:     {"subject": "Your quarterly...", "body": "Hi John,...",
                 "prospect_name": "John Smith", "company": "Acme Corp"}
```

### Phase 4: Settle

The channel is closed. Unused balance is refunded to the user's wallet. A settlement receipt with total spent and invocation count is returned.

### Phase 5: Verify

TEE attestation and receipt verification. The app checks:
1. The attestation was issued by a known enclave platform (AWS Nitro / Intel TDX)
2. The code hash matches the trusted registry
3. The receipt proves output was generated by the attested code
4. The attestation is not expired (default 5-minute TTL)

## SMTP Connector Design

The app does **not** send emails directly. Instead, it provides an **SMTP configuration panel** that connects to the user's existing email infrastructure:

```
┌──────────────────────────────────────────────┐
│           SMTP Connector (config only)        │
│                                              │
│  Server: smtp.sendgrid.net:587               │
│  Username: apikey                            │
│  Password: SG.xxxxx                          │
│  From: sales@acme.com                        │
│  ┌──────────────────────────────────────┐    │
│  │ Test connection     [Save & Verify]  │    │
│  └──────────────────────────────────────┘    │
│                                              │
│  DKIM: ✅ verified for sales.acme.com       │
│  SPF:  ✅ includes smtp.sendgrid.net         │
│  DMARC: ⚠️ p=none (recommend p=quarantine)  │
└──────────────────────────────────────────────┘
```

The SMTP credentials are stored **locally** in the OS keychain (via `flutter_secure_storage` equivalent). They are never transmitted to the marketplace or any third party. The app can:

1. **Validate** SMTP settings (test connection, verify auth)
2. **Check** DNS records (SPF, DKIM, DMARC) via DNS lookup
3. **Score** deliverability readiness
4. **Optionally send** through the configured SMTP relay (user opt-in)

When the user chooses to send through the app, the email is composed locally, checked against marketplace rules, and transmitted directly from the user's machine to the SMTP relay — the marketplace never sees the email content or the recipient address.

## Database Schema

Local SQLite database stores campaigns, templates, and analysis cache:

```sql
-- Campaigns
CREATE TABLE campaigns (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  industry TEXT,
  target_role TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

-- Emails within campaigns
CREATE TABLE emails (
  id TEXT PRIMARY KEY,
  campaign_id TEXT REFERENCES campaigns(id),
  subject TEXT NOT NULL,
  body TEXT NOT NULL,
  structural_hash TEXT,       -- SHA-256 of structural metrics
  word_count INTEGER,
  paragraph_count INTEGER,
  question_count INTEGER,
  link_count INTEGER,
  deliverability_score REAL,
  reply_received INTEGER DEFAULT 0,
  sent_at TEXT,
  created_at TEXT NOT NULL
);

-- Templates (local + marketplace)
CREATE TABLE templates (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  industry TEXT,
  target_role TEXT,
  body_template TEXT NOT NULL,
  source TEXT DEFAULT 'local',  -- 'local' or 'marketplace'
  capability_id TEXT             -- if from marketplace
);

-- Structural analysis cache
CREATE TABLE analysis_cache (
  structural_hash TEXT PRIMARY KEY,
  capability_id TEXT NOT NULL,
  input_json TEXT NOT NULL,
  result_json TEXT NOT NULL,
  invoked_at TEXT NOT NULL,
  price_usd REAL NOT NULL
);
```

## Security Model

### TEE Verification

Every marketplace capability is executed in a Trusted Execution Environment. The app verifies:

```dart
// Before sending data, verify the capability runs in a known enclave
bool verified = agent.verifyTeeAttestation(attestation, capabilityId);
if (!verified) {
  // Do not send data — the capability might not be running expected code
  return;
}

// After receiving results, verify the output was generated in the TEE
bool receiptValid = agent.verifyTeeReceipt(receipt, input, output);
```

### Key Separation

| Key | Storage | Purpose |
|-----|---------|---------|
| Wallet private key | OS keychain | Sign marketplace transactions |
| SMTP credentials | OS keychain | Send emails through configured relay |
| Session tokens | In-memory | Marketplace API authentication |
| Trusted code hashes | Bundled + fetched | Verify TEE attestations |

## Data Flow for a Cold Email Check

```
1. User types email in Composer tab
       │
2. App runs LOCAL structural analysis
       │  word_count, paragraph_count, questions, links
       │
3. User clicks "Score deliverability"
       │
4. App calls AimarketAgent.discover() ─────→ hub.aicom.io
       │  intent: "deliverability rules for cold outreach"
       │  category: "career"
       │
5. App opens channel ─────→ hub.aicom.io
       │  deposit: $5.00 USDT on Base
       │
6. App calls AimarketAgent.invoke() ─────→ hub.aicom.io
       │  capability: selected deliverability checker
       │  input: {industry, target_role, word_count, ...}
       │  ⚠️ No email content or prospect identities
       │
7. Hub executes in TEE ─────→ returns score + recommendations
       │
8. App displays deliverability score + structural recommendations
       │
9. App optionally closes channel
       │  refund received
```
